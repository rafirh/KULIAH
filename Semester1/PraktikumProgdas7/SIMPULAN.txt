Tujuan percobaan untuk merancang dan mengimplementasikan sistem transaksi swalayan dengan konsep OOP telah tercapai sepenuhnya. Sistem berhasil mengenkapsulasi data pelanggan dengan atribut private, mengimplementasikan inheritance melalui class Customer dan turunannya (SilverCustomer, GoldCustomer, PlatinumCustomer), serta memanfaatkan polimorfisme untuk perhitungan cashback yang berbeda-beda tanpa conditional statement. Semua requirement fungsional seperti validasi nomor pelanggan 10 digit, identifikasi tipe berdasarkan prefix, sistem cashback bertingkat, validasi saldo minimum Rp 10.000, autentifikasi PIN, dan pemblokiran akun setelah 3x kesalahan telah diimplementasikan dengan sukses.

Hasil implementasi menunjukkan korelasi kuat dengan dasar teori OOP. Prinsip enkapsulasi terbukti efektif melindungi data pelanggan dari modifikasi langsung sesuai requirement "informasi akun tidak bisa diubah pelanggan secara langsung". Inheritance memungkinkan code reuse dimana method validatePin(), purchase(), dan topUp() diturunkan ke semua subclass tanpa duplikasi kode. Polimorfisme melalui abstract method calculateCashback() memungkinkan TransactionSystem memproses berbagai tipe pelanggan dengan interface yang sama, membuktikan konsep dynamic binding bekerja optimal. Abstraksi melalui abstract class Customer memberikan template yang konsisten untuk semua tipe pelanggan baru yang mungkin ditambahkan di masa depan.

Beberapa faktor teknis mempengaruhi hasil implementasi. Penggunaan HashMap untuk menyimpan pelanggan memberikan performa O(1) pada operasi lookup yang sangat critical untuk transaksi real-time. Scanner buffer clearing dengan nextLine() setelah nextInt()/nextDouble() terbukti esensial untuk mencegah bug input ter-skip. Try-catch exception handling pada input validation membuat program robust terhadap berbagai jenis input error. Sequence validasi yang tepat (cek frozen → validate PIN → cek saldo → execute transaction) memastikan keamanan dan integritas data. Perhitungan cashback sebelum validasi saldo minimum memastikan transaksi valid tidak ditolak secara prematur.

Sistem memiliki beberapa limitasi yang perlu diperhatikan. Akun yang terblokir bersifat permanent tanpa mekanisme unblock, sehingga dalam implementasi production perlu ditambahkan fitur reset atau customer service intervention. Data pelanggan disimpan in-memory tanpa persistensi ke database, sehingga hilang saat program ditutup. Validasi nomor pelanggan hanya mengecek panjang 10 digit tanpa checksum atau format validation yang lebih strict. PIN disimpan sebagai plain text tanpa enkripsi yang dalam production environment merupakan security vulnerability. Tidak ada mekanisme session timeout atau rate limiting untuk mencegah brute force attack. Error handling sudah cukup robust namun masih menggunakan generic Exception yang sebaiknya diganti dengan specific exception types untuk debugging yang lebih baik.

Sistem transaksi swalayan Tiny berhasil diimplementasikan dengan menerapkan prinsip-prinsip OOP secara komprehensif. Enkapsulasi, inheritance, polimorfisme, dan abstraksi bekerja sinergis menghasilkan kode yang modular, maintainable, dan extensible. Sistem mampu menangani semua skenario transaksi yang direquire dengan validasi dan error handling yang memadai. Meskipun terdapat beberapa limitasi untuk production deployment, implementasi ini berhasil mendemonstrasikan penerapan OOP dalam membangun sistem transaksional yang aman dan reliable untuk keperluan pembelajaran.
